# Visual Explanation Shot Lab 01: Prompt Completion Prototype

**日期**: 2026-06-18
**Composition**: `PromptCompletionExplainer` (630 frames / 21s / 30fps)
**位置**: `src/video-system/experiments/shot-lab/PromptCompletionExplainer.tsx`
**Studio ID**: `PromptCompletionExplainer`

---

## 原型目标

验证"画面本身参与讲解"的视觉解释镜头——用动画演示 AI 提问从模糊到具体的过程，而非用文字卡片陈述观点。

## 画面设计

### 5 阶段时间轴

| 阶段    | 帧范围  | 时长 | 画面内容                                            |
| ------- | ------- | ---- | --------------------------------------------------- |
| Phase 1 | 0-90    | 3s   | 左侧空泛问题框"帮我写一下" + 右侧灰雾模糊回答       |
| Phase 2 | 90-270  | 6s   | 四个上下文模块（身份/目标/限制/输出）依次飞入问题框 |
| Phase 3 | 270-390 | 4s   | 问题框拼图补齐，闪光效果                            |
| Phase 4 | 390-540 | 5s   | 灰雾散开，四段结构化回答揭示                        |
| Phase 5 | 540-630 | 3s   | 全屏结论："AI 没变，材料变完整了"                   |

### 动画技术

- **spring()** — 所有元素出现、飞入、发光用 spring 物理动画
- **interpolate()** — 灰雾散开、进度过渡、淡入淡出
- **Math.sin()** — 灰雾呼吸脉动、问题框微抖
- 全程 frame-driven，零 CSS transition/animation

### 视觉元素

| 元素                 | 作用               | 动画方式                          |
| -------------------- | ------------------ | --------------------------------- |
| VagueQuestionBox     | 模糊提问，虚线边框 | spring 弹入 + 微抖暗示不完整      |
| FoggyAnswer          | 灰色模糊回答区     | radial-gradient + blur + 呼吸脉动 |
| ContextModule x4     | 四个上下文标签     | 从左下飞入，旋转归位，到达后发光  |
| CompletedQuestionBox | 补齐后的完整提问   | 渐变边框 + 闪光                   |
| StructuredAnswer     | 四段结构化结果     | blur 渐消 + 逐条滑入              |
| ConclusionText       | 结论叠加           | spring 放大出现                   |

## 验收报告

### Q1: 是否比当前卡片式画面更像"画面在讲解"？

**是，有明显区别。** 当前卡片式画面（如 bullets、comparison）本质是"把要点排版在屏幕上"，观众读完就完了。这个原型的核心差异是：**观众看到的是一个变化过程**——模糊问题 → 补充上下文 → 拼图补齐 → 回答清晰化。画面的"动"在传递因果关系（材料变完整 → 结果变清晰），而非装饰。

### Q2: 哪些元素还像 PPT？

1. **结论文字页（Phase 5）**— 全屏大字 + 副标题，本质就是 PPT 的 ending slide
2. **四段结果列表**— 虽然有逐条滑入动画，但"图标 + 标题 + 序号"的排列方式仍然是信息卡片
3. **Phase 标签**— 左上角的阶段提示是演示文稿思维，正式视频中不应出现

### Q3: 哪些动画真正帮助理解？

| 动画                 | 是否帮助理解 | 原因                                       |
| -------------------- | ------------ | ------------------------------------------ |
| 灰雾模糊 → 渐渐清晰  | **是**       | 直观展示"信息不足 = 结果模糊"的因果关系    |
| 上下文模块飞入问题框 | **是**       | 观众看到"材料在填充"，理解补齐的过程       |
| 拼图补齐闪光         | **部分**     | 闪光偏装饰，但"虚线变实线"的边框变化有意义 |
| 四段结果逐条滑入     | **弱**       | 仍然是列表展示，没有解释为什么是这四段     |
| 结论文字放大         | **否**       | 纯强调，不帮助理解                         |

### Q4: 是否值得沉淀为 Visual Explanation Shot Pattern？

**值得，但需要提炼核心 pattern。** 这个原型验证了两个有效的视觉解释手法：

1. **Before/After 因果动画**：左侧"因"变化 → 右侧"果"同步变化。灰雾散开是最成功的视觉隐喻。
2. **拼图补齐**：不完整的输入 → 模块飞入补齐 → 完整输入。这个过程本身就是解释。

可以抽象为 `CauseEffectPair` pattern：一个输入区 + 一个输出区，输入变化驱动输出变化。

### Q5: 是否可以作为后续 AI 提问类视频的稳定候选资产？

**可以作为候选，但不是即插即用。** 需要解决：

- **内容泛化**：当前硬编码了"帮我写一下"的具体场景，需要参数化
- **视觉风格**：暗色极光风适合科技话题，其他主题需要适配
- **结果展示**：四段列表太平，考虑用更动态的方式展示结构化输出（如拼图块逐个亮起）

## 技术债务

- 子组件内联样式过多，正式化时需提取为 styled components 或 design tokens
- ContextModule 的飞入目标坐标硬编码，不同内容长度需要自适应
- FoggyAnswer 的 blur 效果在低端设备可能有性能问题
