# 视觉设计哲学完整版

> 完整定义、边界、例子和反例。供遇到疑难设计、能力冲突和审片争议时查阅，不要求每次完整通读。

---

## 1. 解释、理解、吸引三类贡献

### 1.1 解释贡献

画面让观众看到口播无法单独传达的信息。

有效例子：

- 流程图展示步骤之间的先后和分支关系；
- 对比画面展示两条路径的差异；
- 代码 diff 展示修改前后的具体变化；
- 截图展示真实工具界面的实际状态。

无效例子：

- 口播说"有三个要点"，画面列出三个标题——信息完全重复，画面无独立贡献；
- 口播说"这个方法更好"，画面只显示一个绿色对勾——标签冒充解释。

### 1.2 理解贡献

画面降低认知负荷。

有效例子：

- 将抽象概念映射为具体视觉隐喻；
- 用颜色区分正面和负面内容；
- 用空间位置表达层次关系；
- 用动画展示变化过程而非只展示结果。

无效例子：

- 装饰性背景图；
- 与内容无直接关系的插画；
- 只为"好看"而存在的视觉元素。

### 1.3 吸引贡献

画面产生继续观看的意愿。

有效例子：

- Hook 中的冲突画面制造好奇；
- 状态阻断制造"然后呢"的悬念；
- 节奏变化打破单调；
- 关键转折处的视觉强调。

无效例子：

- 每个页面都用相同的入场动画——重复后失效；
- 无信息量的动态背景——分散注意力而非制造吸引。

### 1.4 无贡献占位画面

以下特征组合出现时，判定为无贡献占位：

- 三项贡献都弱；
- 删除后观众理解不受影响；
- 画面内容可以被口播完全替代；
- 只有形式（标题、卡片、入场动画）没有语义。

## 2. 有效与无效的声画重复

### 2.1 有效重复

声画传递相同信息但各自承担不同劳动：

- 口播说"代码从 200 行缩减到 50 行"，画面展示具体的 200 行和 50 行代码——信息方向一致，画面提供具体证据；
- 口播说"两条路径都能到达目标"，画面展示两条路径的空间差异——信息方向一致，画面提供空间理解。

### 2.2 无效重复

声画传递完全相同的信息，画面无独立劳动：

- 口播说"第一步是打开设置"，画面显示文字"第一步：打开设置"——纯文字复读；
- 口播说"结果出乎意料"，画面显示"出乎意料！"——感叹号不构成视觉贡献。

### 2.3 判断标准

问一个问题：如果关掉画面只听声音，观众会丢失什么？

- 丢失具体证据 → 有效；
- 丢失空间关系 → 有效；
- 丢失视觉体验 → 有效；
- 什么都没丢失 → 无效重复。

## 3. 连续性类型

### 3.1 对象连续性

同一视觉对象跨 Shot 持续存在。

例子：CLAUDE.md 文档从第一个 Shot 开始出现，到结尾逐渐增长——同一对象，状态变化。

反例：每个 Shot 都重新画一个"新"的 CLAUDE.md——对象身份断裂。

### 3.2 状态连续性

对象的状态变化是累积的，不是每个 Shot 独立重置。

例子：文档从空白 → 有标题 → 有规则 → 完整——状态弧线。

反例：文档在 Shot 3 是空白，Shot 4 突然变成完整——状态跳跃。

### 3.3 因果连续性

后一个视觉事件是前一个的结果。

例子：用户提出问题 → 系统给出错误答案 → 用户纠正 → 系统给出正确答案——因果链。

反例：错误答案和正确答案同时出现，没有纠正过程——因果缺失。

### 3.4 问题连续性

Hook 提出的问题在全片中逐步被回答。

例子：Hook 问"为什么 Claude Code 每次都像第一天？"→ 中段展示具体表现 → 后段展示解决方案 → 结尾回收问题。

反例：Hook 问了问题，中段转向不相关的话题——问题断裂。

### 3.5 情绪连续性

情绪弧线有起伏，不是全程一个调。

例子：好奇 → 困惑 → 豁然开朗 → 满足——情绪变化。

反例：全程"平稳讲述"——无情绪弧线。

### 3.6 视觉母题连续性

一个视觉元素在全片中反复出现，每次承载递进含义。

例子：一个锁的图标在"锁定规则"场景出现，后来在"自由发挥"场景中被打开——同一母题，含义变化。

反例：视觉元素随机出现，无递进关系。

## 4. 五类视觉模式及选型方法

### 4.1 visual-narrative（视觉叙事）

让观众经历一个过程。

适用：Hook、案例展示、状态变化、冲突与解决。

选型条件：内容有时间线、因果链或状态变化。

### 4.2 visual-explanation（视觉解释）

让观众看懂一个关系或机制。

适用：流程、对比、结构、原理。

选型条件：内容有空间关系、层次结构或并行对比。

### 4.3 editorial-design（编辑设计）

让观众看清信息结构。

适用：列表、要点、数据、总结。

选型条件：内容是信息集合，需要清晰组织。

### 4.4 typographic-performance（文字表演）

让观众感受到语气和节奏。

适用：关键判断、反转、情绪强调。

选型条件：内容的情感重量需要通过文字的空间运动来传达。

### 4.5 real-evidence（真实证据）

让观众相信事实。

适用：截图、代码、数据、实验结果。

选型条件：内容需要真实材料支撑，不能用抽象图形替代。

### 4.6 选型规则

- 每个关键 Shot 一个主模式，最多一个辅助模式；
- 不得把模式名称直接绑定成唯一 Scene 组件；
- 选型依据是表达需求，不是组件可用性。

## 5. 戏剧化、隐喻和证据的真实性边界

### 5.1 戏剧化

允许放大体验，不允许放大结论。

有效：用紧张的节奏展示用户遇到 bug 时的焦虑——放大体验。

无效：把一个普通 bug 表现成系统性灾难——放大结论。

### 5.2 隐喻

允许借隐喻解释，不允许用隐喻代替证明。

有效：用"迷宫"隐喻解释复杂的配置流程——帮助理解。

无效：用"迷宫"隐喻代替具体的配置步骤说明——代替证明。

### 5.3 证据

允许设计证据的展示方式，不允许改变证据本身。

有效：高亮代码中的关键行，放大显示——设计展示。

无效：修改代码内容使其看起来更简洁——改变证据。

### 5.4 边界判断

问一个问题：如果观众事后核实，会发现画面说谎了吗？

- 不会 → 安全；
- 可能会 → 需要审查；
- 一定会 → 禁止。

## 6. 设计理由与可迁移表达

### 6.1 设计理由

每个关键 Shot 必须说明：

- 为什么需要这个画面；
- 想改变观众什么认知；
- 为什么选择这种视觉模式；
- 哪些意义不能丢失。

设计理由不是"为了让观众看到口播内容"。

### 6.2 可迁移表达

设计理由应该可以迁移到不同实现方式。

好的设计理由："展示两条路径的空间差异，让观众直观理解为什么直接路径更好"——即使换成不同的组件，这个目标仍然成立。

坏的设计理由："使用 ComparisonScene 组件"——这是实现方式，不是设计理由。

---

## 7. Component-Blind / Intent-First 设计

### 7.1 创意阶段对组件失明

在定义视觉意图时，不得考虑现有组件能做什么。

先定义：

- 观众看到什么；
- 初始状态是什么；
- 核心视觉事件是什么；
- 状态怎样改变；
- 结果是什么。

### 7.2 实现阶段对能力清醒

视觉意图锁定后，才允许读取：

- capabilityRegistry；
- Scene 类型；
- 现有组件；
- 素材能力；
- Remotion 实现约束。

### 7.3 禁止反向锚定

现有组件可以决定怎么做，不能决定做什么。

如果视觉意图无法用现有组件实现，应发起能力谈判，而不是降低视觉目标。

## 8. 能力谈判与 L0—L3 降级等级

### 8.1 降级等级

- **L0**：仅形式变化，无实质损失。可自行决定。
- **L1**：轻微表现力损失，必须记录。可自行决定但需记录。
- **L2**：解释、理解或吸引明显下降，必须获得用户批准。
- **L3**：语义或事实损失，禁止执行。

### 8.2 谈判流程

1. 确认原始视觉意图和认知目标；
2. 识别能力差距；
3. 评估现有组件为何无法满足；
4. 提出降级选项，标注降级等级；
5. L0-L1 自行决定并记录；
6. L2 提交用户批准；
7. L3 不作为可执行方案。

### 8.3 禁止事项

- Agent 无权自行批准 L2；
- L3 不得作为可执行方案；
- 不得静默降级——任何降级必须记录。

## 9. 五级完成状态

### 9.1 层级定义

1. **Not Implemented**：代码未编写。
2. **Implemented, Unverified**：代码存在但未验证。
3. **Technically Verified**：代码运行无报错。
4. **Visually Verified**：实际渲染画面符合设计意图。
5. **Semantically Verified**：画面表达了正确的语义。
6. **Viewer Accepted**：观众实际观看后确认理解。

### 9.2 硬规则

只有 `Viewer Accepted` 才能称为视觉完成。

命令成功不等于视觉通过。测试通过不等于语义正确。

## 10. Shot、Beat、全片三级验收

### 10.1 Shot 级

- 画面是否表达了 lockedMeaning；
- 三类贡献是否明确；
- 声画关系是否有效；
- 是否有无贡献占位。

### 10.2 Beat 级

- Beat 内各 Shot 是否形成连贯的认知变化；
- viewerStateBefore → viewerStateAfter 是否实现；
- 信息增量是否持续。

### 10.3 全片级

- Hook 承诺是否兑现；
- 核心问题是否被回答；
- 情绪弧线是否完整；
- 结尾是否回收。

---

## 11. 冷观看与设计理由对照

### 11.1 冷观看

不看设计文档，只看实际画面，判断：

- 观众能看懂吗；
- 观众会继续看吗；
- 观众会相信吗。

### 11.2 对照

冷观看结论与设计理由对比：

- 如果冷观看结论与设计意图一致 → 通过；
- 如果冷观看结论与设计意图不一致 → 需要审查是实现问题还是设计问题。

---

## 12. 自动测试的能力边界

自动测试可以验证：

- 技术正确性（代码运行无报错）；
- 布局约束（裁切、溢出、字号）；
- 数据完整性（必填字段存在）；
- 时序正确性（帧数、时长）。

自动测试不能验证：

- 语义正确性（画面是否表达了正确的含义）；
- 理解效果（观众是否看懂）；
- 吸引效果（观众是否愿意继续看）；
- 占位判断（画面是否只是填充）。

这些必须通过实际渲染和人工审查验证。

---

## 13. 真实渲染证据要求

不同问题需要不同类型的证据：

| 问题类型 | 证据形式                        |
| -------- | ------------------------------- |
| 裁切     | 实际帧截图 + 边界标注           |
| 动态事件 | 前、中、后三帧或短片段          |
| 状态继承 | 相邻 Shot 边界截图              |
| 可读性   | 手机端缩放截图 + 正常播放       |
| 节奏     | 连续片段播放                    |
| 全片重复 | contact sheet + 统计 + 完整观看 |

命令输出、测试报告、Agent 自报状态不构成视觉证据。

---

## 14. 否决项

以下任一出现，禁止发布：

- 画面语义与口播相反；
- 因果方向错误；
- 核心视觉事件缺失；
- 连续状态被重置冒充为持续变化；
- 隐喻代替证据；
- 修改证据原意；
- 核心信息在实际设备不可见；
- 未经批准的 L2 降级；
- 任何 L3 降级；
- Agent 以测试通过或渲染成功宣称视觉完成。
