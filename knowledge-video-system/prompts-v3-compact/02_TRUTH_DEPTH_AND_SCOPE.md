# 事实、深度与范围

## 目标

回答一个问题：内容是否准确、可信、完整，而且范围合理？

本文件是第 1、2 阶段的必要质量约束，不得弱化。

## 核心判断

### 1. 事实与推断区分

- 哪些是官方文档、真实实验、可验证的事实？
- 哪些是个人经验、推断、观点或比喻？
- 推断和观点是否明确标注，不冒充事实？

### 2. 证据充分性

- 核心结论有没有真实案例或实验支撑？
- 能否给出具体场景、操作步骤或前后对比？
- 没有证据的结论必须标注为"推测"或"假设"

### 3. 核心结论

- 这条视频的核心结论是什么？
- 结论是否经得住反例检查？
- 结论是否太绝对，需要加限定词？

### 4. 观众必问问题

- 观众看完最可能问什么？
- 这些问题在正文中是否已经回答？

### 5. 反例和边界

- 什么情况下这个结论不成立？
- 是否需要说明适用范围？

### 6. 内容完整度

- 标题承诺的问题是否完整回答？
- 范围是否过大，应该拆分？

### 7. 因果边界

- 不把一次案例、一次实验或相关性扩大成普遍因果
- 单次实验只能说"我试了一次"，不能说"这样做一定有效"

### 8. 搜索触发规则

必须搜索或核验：

- 当前软件和产品能力
- 官方规则、平台政策
- 数字、比例、时间、价格、版本
- 法律、医疗、金融、安全
- 竞品比较
- 强因果结论
- 可能已经变化或模型不确定的事实

可以直接基于用户材料或稳定知识：

- 用户真实经历和实验
- 已上传并可验证的证据
- 稳定基础概念
- 纯逻辑推理
- 创意、叙事和视觉设计

## Standard Evidence Notes

Standard 模式输出简化证据笔记，至少包含：

```text
claim: [主张]
evidenceType: [official / experiment / inference / opinion / analogy]
source: [来源]
sourceDate: [日期]
confidence: [high / medium / low]
allowedWording: [可以怎么说]
boundary: [什么情况下不成立]
```

涉及当前软件行为、官方规则、产品能力、平台政策和可能变化的信息，应搜索并优先使用官方或一手来源。搜索是为了支撑视频结论，不得把视频变成资料堆积。

## 模式差异

### Quick

- 只检查关键事实是否可信
- 不生成完整 Evidence Notes
- 对明显事实错误直接指出

### Standard

- 输出 Standard Evidence Notes
- 标注哪些是事实、哪些是推断
- 检查反例和边界

### Deep

- 调用 09 生成完整 Scope Contract / Coverage Map / Fact Evidence Map
- 启用多 AI 独立审查
- 要求 90 分门禁

## 真实性边界对画面系统的约束

事实审查结论必须传递给画面系统，作为视觉设计的硬约束：

- 事实、实验、推断、观点和类比必须区分——画面不得模糊这个区分
- 单次实验不能自动扩展为普遍因果——画面不得把个案表现成普遍结论
- 戏剧化只允许放大体验，不允许放大结论
- 隐喻只允许帮助理解，不允许代替证明
- 证据只允许设计展示方式，不允许改变证据本身

## 输出

```text
核心结论：[一句话]
事实基础：[支撑结论的关键事实]
推断标注：[哪些是推断，为什么可接受]
反例/边界：[什么情况下不成立]
因果边界：[单次实验 vs 普遍规律]
完整性：标题承诺是否兑现
truthBoundaryForVisual: [画面系统不得越过的事实边界]
建议：通过 / 需要补充 / 需要拆分
```
